2 · Anatomia del system prompt

El prompt es munta a server.py:1202 build_system_prompt(). L'ordre real dels blocs és:

  1. Identitat (capa 1, fixa)
  2. Instruccions filtrades (capes 2-3, dinàmiques per perfil/MECR/DUA/complements)
  3. Bloc DUA (corpus M2)
  4. Bloc gènere discursiu (si especificat)
  5. Creuaments (si 2+ perfils actius)
  6. PERSONA-AUDIENCE (narrativa concreta de l'alumne)
  7. Context RAG (actualment buit — RAG-KG desactivat el 2026-04-09, veure server.py:2349)
  8. FORMAT DE SORTIDA + complements (instruccions detallades per a Glossari, Preguntes MALL/TILC, Bastides, etc.)

2.1 IDENTITAT

Activa-se sempre. És un string fix definit a corpus_reader.py:61 dins de load_corpus() (no es llegeix de cap fitxer markdown).

Codi real (corpus_reader.py:60-80) 
Ets l'assistent ATNE (Adaptador de Textos a Necessitats Educatives) de Jesuïtes Educació.

OBJECTIU: Transformar textos educatius perquè siguin accessibles a l'alumnat descrit, seguint principis de DUA, Lectura Fàcil i MECR.

TO: Acadèmic neutre. Respecta el registre del text original quan sigui identificable.

FIDELITAT:
- En adaptació: cada element ha de tenir correspondència amb l'original (Mayer). No inventis dades, exemples ni fets.
- En complements (glossari, preguntes, esquemes): crea contingut nou derivat del text adaptat.

RIGOR TERMINOLÒGIC: Conserva sempre els termes curriculars. Defineix-los, no els eliminis. MAI substitueixis per parafrasis buides ("la cosa verda", "el que fa que", "un tipus de").

FORMAT: Comença DIRECTAMENT amb "## Text adaptat". Zero meta-text ("Here is...", "Let me...", "Okay...").

LLENGUA: Català (o la llengua vehicular indicada).

SEGURETAT:
- No reprodueixis dades personals de l'alumne al text adaptat.
- Evita exemples potencialment traumàtics amb perfils vulnerables.
- No inventis estadístiques, dates ni fets no presents al text original.

Els IDs no es mostren al LLM: queden a logs/audit per a traçabilitat.

Exemple real — bloc «LÈXIC» del cas TDAH (10 primeres línies) 
**LÈXIC**:
- Usa vocabulari freqüent. Substitueix termes poc habituals per equivalents d'alta freqüència lèxica.
- Termes tècnics en **negreta** amb definició entre parèntesis la primera vegada.
- Repetició lèxica coherent: un terme = un concepte. NO variïs per elegància (no sinònims).
- Referents pronominals explícits: si ambigu, repeteix el nom complet.
- Elimina expressions idiomàtiques, metàfores i sentit figurat. Tot literal.
- Controla polisèmia: evita usos figurats o poc habituals d'un mot. Permet sentits habituals.
- Connectors explícits entre frases: per tant, a més, en canvi, primer, després.
- Scaffolding lleuger: defineix un terme la 1a vegada; després usa'l sense definició.
- Desnominalitza: noms abstractes → verbs. Exemple: 'l'evaporació' → 'quan s'evapora'.
- Evita doble negació. Permet negació simple i natural.
- Dates en format complet (12 de març de 2026, no 12/03/26). Xifres amb context.
- Sigles i abreviatures: escriu la forma completa la primera vegada. Ex: ONU (Organització de les Nacions Unides).

**SINTAXI**:
- Una idea per frase. Divideix frases llargues en unitats simples.
- Prefereix veu activa. Permet passiva quan sigui natural i clara.
- Subjecte explícit quan hi hagi risc d'ambigüitat. Permet elisió en contextos clars.
- Ordre SVO per defecte. Permet inversions estilístiques si no generen ambigüitat.
- Puntuació estàndard simplificada. Permet punt i coma ocasional. Evita parèntesis llargs.
- Màxim 12-18 paraules per frase.
- Es permeten subordinades simples (que, quan, si).
- Permet incisos breus i parèntesis explicatius. Evita incisos de >8 mots.

**ESTRUCTURA**:
- Paràgrafs curts: 3-5 frases màxim. Un tema per paràgraf.
- Blocs temàtics amb títol descriptiu. Format pregunta quan sigui possible.
- Frase tòpic al principi de cada paràgraf: anticipa el contingut.
- Llista si 4+ elements o si l'enumeració és complexa.
- Estructura deductiva: general → particular. Primer la idea, després els detalls.

4 · Fases del LLM (seqüencials)

Quan el frontend crida POST /api/adapt, el handler (server.py:4346) llança run_adaptation() en un ThreadPoolExecutor i emet events SSE. Aquesta és la seqüència real:

4.1 Phase 1 — adapt

  • Resol model: _model_for("adapt", override) (server.py:158) — pot ser fix o rotate.
  • Construeix system prompt: build_system_prompt(profile, context, params, rag_context="") a server.py:2364.
  • Crida LLM via _call_llm() (provider-specific: gemma4/gemini/mistral/gpt/openrouter).
  • Neteja: clean_gemini_output() + _post_process_llm_output() (LaTeX strip, anglès→català, typos, concatenacions).
  • SSE event tipus step a cada subpas ("Generant adaptació amb {model}...").

4.2 Phase 2 — verify

  • Activa-se per defecte (params.verify_retry = true). Es pot desactivar.
  • Crida _verify_adaptation(): prompt minimal (VERIFY_SYSTEM) que demana JSON {Q,P,C,j} (Qualitat, Pedagogia, Coherència 0-5).
  • Si avg < 4.0, dispara retry: torna a Phase 1 (màx 2 intents). Es queda amb la millor versió.

4.3 Phase 3 — quality

Pipeline determinista de qualitat català (post_process_catalan()). 4 capes + 1 opt-in:

  1. Caràcters exòtics — _exotic_char_scan()
  2. LanguageTool — _languagetool_full_analysis() (API pública, determinista)
  3. Diccionari + paraules sospitoses
  4. Llegibilitat — _readability_score() per MECR target
  5. Auditor LLM (opt-in, DESACTIVAT per defecte des del 14/04) — _llm_audit(). Falsos positius ~85% en proves.

4.4 Phase 4 — parseAdaptedSections (frontend)

El text retornat per Phase 1 té la forma ## Text adaptat ... ## Glossari .... La funció parseAdaptedSections() separa el text per ##, normalitza els títols (sense accents, lowercase, sense puntuació) i els mapeja a claus internes (_main, glossari, esquema_visual, mapa_conceptual, preguntes_comprensio, bastides, ...). El Pas 3 distribueix cada secció al panel corresponent (applyComplementsToViews()).

4.5 Refine — POST /api/refine-text

Endpoint independent (server.py:4207) que NO regenera des de zero — modifica el text adaptat seguint un preset o instrucció lliure. Veure detall a la secció 3.4.

8.4 Com s'aplica al prompt d'adaptació (la pregunta clau)

El to NO es propaga a l'adaptació.
  • El payload de /api/adapt no conté mai params.to ni context.to — verificat a llm.js:125-129 i pas3.html:1915.
  • build_system_prompt() no llegeix cap variable de to — verificat amb grep: l'única referència a params.to al backend és zero.
  • El bloc IDENTITAT (capa 1) diu: TO: Acadèmic neutre. Respecta el registre del text original quan sigui identificable. (corpus_reader.py). És l'única «pista» de to que rep l'LLM al pipeline d'adaptació.
  • La instrucció G-06 (Ajusta el to segons MECR) emet variants de to en funció del nivell MECR — no en funció de la selecció del docent.

8.5 Conseqüència i hipòtesi

  • Conseqüència: si el docent genera un text motivador amb to «motivador» i després l'adapta per a un perfil, l'LLM d'adaptació no rep aquesta instrucció. Només percep el to del text que li passen (heurística implícita) i la pauta G-06 modulada per MECR.
  • Hipòtesi: part del to es manté per inèrcia — l'LLM tendeix a conservar el registre del text original que ha de transformar. Però cap instrucció explícita ho garanteix.
  • Marca: GAP documentat — no és funcionalitat d'adaptació en producció. Si el pilot demana «adaptació amb to X», caldria: (a) afegir selector de to al Pas 3 o propagar el del Pas 2; (b) afegir params.to al payload de /api/adapt; (c) afegir una capa «TO DEMANAT» a build_system_prompt abans del PERSONA-AUDIENCE.

10 · Persona-audience — aporta valor?

Pregunta oberta — cal experiment.

10.1 Què és

La funció build_persona_audience() transforma les característiques del perfil en una narrativa en català natural. En lloc de passar {"tdah": {"grau": "moderat"}} a l'LLM, emet «TDAH, presentació combinat (grau moderat). Memòria de treball baixa: limitar elements simultanis, repetir informació clau.» (Exemple del Cas A, secció §5.1).

Cobreix 13 perfils amb les seves sub-variables: nouvingut (L1, alfabet, escolarització, CALP), TEA (DSM-5, comunicació oral), TDAH (presentació, grau, memòria, fatiga), dislèxia (tipus, grau), altes_capacitats (global vs talent específic), DI (grau), TDL (modalitat, grau, sub-afectacions), disc. visual/auditiva, trastorn emocional, vulnerabilitat, TDC, 2e.

10.2 Per què es podria pensar que és redundant

  • Les instruccions H-* ja descriuen què fer per cada perfil. Si l'LLM ja rep «TDAH: micro-blocs de 3-5 frases amb objectiu explícit» (H-04), ¿necessita a més llegir «Memòria de treball baixa: limitar elements simultanis»?
  • Les sub-variables ja disparen intensificacions automàtiques a _get_intensified_text(). Pot ser sorollós tenir dos senyals (narratiu + instrucció modulada) per al mateix fet.
  • El prompt s'allarga 200-400 caràcters per cada perfil. Amb quotes de tokens limitades, això compta.

10.3 Per què podria aportar valor

  • Els LLMs són entrenats majoritàriament amb llenguatge natural. Una descripció narrativa de «per a qui» escrius pot activar representacions internes més riques que una llista de regles.
  • Gemma 3 27B i Gemma 4 31B (els models pilot) responen bé a persona prompting (evidència empírica multi_v2, sessió 31/03).
  • En perfils compostos (2e, grup mixt) la narrativa permet matisos que les instruccions atòmiques no capturen: «TDAH + AACC = repte alt amb suport de format».

10.4 Hipòtesi

Hipòtesi treballable: alguns models es beneficien de la persona-audience (segurament els chat-tuned com GPT-4o, Gemma 4), altres la tracten com a soroll (possiblement Mistral, models més instructius). No tenim evidència empírica neta en aquest repo.

  • Experiment A/B · 4 models actius (Gemma 3 27B, Gemma 4 31B, GPT-4o, GPT-4.1-mini) × 20 casos × 2 condicions (amb/sense persona) = 160 adaptacions.
  • Avaluar amb la rúbrica evaluator_rubrics.py (5 dimensions × 1-5) + HITL cec (3 docents per cas).
  • Mesurar també: paraules del prompt, latència, cost per crida. Si persona no aporta però encareix, decisió fàcil.
